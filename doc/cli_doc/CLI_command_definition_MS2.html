Open/SpeedShop Command Line Definitions

DEFINITIONS:

  • Experiment: A set of collectors and executables bound together to generate performance metrics..
  • Collector: Portion of the tool containing logic that is responsible for the gathering of the performance metric.
  • Metric: The entity, which the collector/experiment is gathering. A time, occurrence counter, or other entity, which reflects in some way on the applications performance and is gathered by a performance experiment (by the collector).
  • Focused experiment:  The experiment that is in current active context.  The last created experiment is usually the default or focused experiment, however, the expFocus command will set the experiment specified in the expFocus command to be the focused or default experiment.
  • Framework:  The set of API functions that allows the user interface to manage the creation and viewing of performance experiments.  It is the interface between the  user interface and the DPCL and Dyninst components.
    •

ASSUMPTIONS USED IN DEFINITION OF COMMANDS:

  1. There is a focused (default) experiment id available to subsequent commands.
    2.
  2. Our tool will not change the focus of the tool on an experiment. It can only be changed or set initially by the user. The one exception to this is when the experiment is created via the expCreate command the focus is set to this experiment.
  3. The user can change the focus at anytime via the expFocus command.
  4. Commands with no explicit experiment id refer to the currently focused experiment, e.g. the default experiment.
  5. There is no experiment context (default) until expCreate command is successfully completed.
  6. expCreate  with no application/collector defines an experiment and returns and experiment id. Collectors/precanned experiments can be added later. More application objects can also be added after the expCreate command is completed.
    7.

Command detailed definition section
 
  --------------------------------------------

EXPERIMENT BUILDING BLOCK COMMANDS
 

  • expAttach 
      - Attach application information or collectors to an experiment.

      The -f filename option can be used to guarantee the rank, process, or thread given or when the rank, process, or thread would be ambiguous from one system to another. In that case, all occurrences of this filename will be attached to the current performance measurements.

      -f filename: Optional fully or partially qualified symbol name denoting the executable file corresponding to the target program.

      -r, -p, -t options: Rank/Process/Thread id to attach to the performance tool.

      Examples:

      expAttach -h baguette -p 1234
      - attaches to the pid 1234 on baguette.engr.sgi.com

      expAttach -h baguette -f /opt/SpeedShop/demos/generic/generic.exe
      - attaches to all occurrences of generic.exe running on baguette.engr.sgi.com

      expAttach -h baguette -f /opt/SpeedShop/demos/generic/generic.exe -r 1
      - attaches to rank 1 of generic.exe running on baguette.engr.sgi.com

      expAttach -f generic -p 1234
      - attaches to all threads of application generic which pid is 1234.

      expAttach [ <expId_spec> ] [ <host_file_rpt_list> ] [ <expType_list> ]
  • expClose
      - Close/Terminate the experiment specified by experiment id or if none specified close the default experiment (focused experiment). expClose releases the experiment associated storage and also removes the internal experiment data files. expClose also takes an optional all parameter which when specified indicates the user?s intent to close all experiments.

      expClose [ <expId_spec> || all ]
  • expCont 
      - Continue the experiment specified by experiment id expCont also takes an optional all parameter which when specified indicates the user?s intent to continue all experiments.

      expCont [ <expId_spec> || all ]

  • expCreate 
      • - Start the process of defining an experiment:

    • define a new experiment identifier,
    • set the current focus to the new identifier and
    • return the experiment identification identifier.
      •
      - The experiment type argument "<expType_list>" can be used to specify an experiment and determines the default values that will be used in the definition of the experiment. 
      -The <host_file_rpt_list> will associate the specified executable with the experiment being defined.
      - This command also sets the focus to the new experiment?s identifier. There can be a list of application objects and a list of collectors associated with an experiment.

      <expId> = expCreate [ <host_file_rpt_list> ] [ <expType_list> ]

  • expDetach 
      - detach application information or collectors from an experiment.

      expDetach [ <expId_spec> ] [ <host_file_rpt_list> ] [ <expType_list> ]
  • expFocus 
      - Make the input experiment id the active experiment. It becomes the current or default experiment.
      - If no argument is present, return the focused experiment.
      - If the expId_spec argument has a value of zero (0), then clear the experiment focus.

      [ <expId> = ] expFocus [ <expId_spec> ]
  • expPause
      - Temporarily halt the experiment identified by the input experiment id.

      expPause [ <expId_spec> || all ]
  • expRestore
      - Restore an experiment's definition and the previously saved performance data from a file for further analysis.
      - A new experiment is created to contain this data.
      - This command also sets the focus to the new experiment?s identifier.

      <expId> = expRestore <host_file>
  • expRun
      - Run the experiment and collect performance data.

      expRun [ <expId_spec> || all ]
  • expSave
      • - Save the experiment performance data from a file for further analysis.
      - The use of the all option will cause the application to be saved in the specified file, along with the data that was collected about it.
      - The use of the data option will cause only the data that was collected about the application to be saved in the specified filed. When this data is used for additional analysis, the user will need to make the original application available.

      expSave [ <expId_spec> ] { all || data } <host_file>
  • expSetParam 
      - Add an experiment argument to the focused experiment. <expParam> will be unique within the experiment. The parameter names can be used for other experiments.

      expSetParam [ <expId_spec> ] <expParam> = <expParamValue> [ , <expParam> = <expParamValue> ] ...
  • expStop
      - Stop the experiment specified by experiment id and remove instrumentation.

      expStop [ <expId_spec> || all ]
  • expView
      - View the data associated with a particular <viewType> for the experiment specified by <expId_spec>.
      - The optional -gui argument will cause the data to be directed to an appropriate window for display in a graphical format.

      expView [ <expId_spec> ] [ -gui ] <viewType>

    INFORMATION COMMANDS

  • listExp
      • - List the experiments that can be run, along with detailed information about the data measurement utilities and applications that may have been attached to the experiment.
      - The absence of any experiment specification will cause all the experiments that have been defined by the user, through the use of expCreate commands, to be listed.
      - The "focus" option will cause only the focused experiment to be listed.
      - The <expId_spec> option will cause the selected experiment to be listed.
      - The "all" option will cause all the available performance measurement utilities, that can be used to define experiments, to be listed.
      - The output will be similar to that of the listing of breakpoints in gdb, dbx, cvd, or totalview.

      <ListOf_string> = listExp [ focus || <expId_spec> || all ]
  • listHosts
      • - Lists the hosts that define the specified cluster.
      - The default cluster contains only localhost.

      <ListOf_hostname> = listHosts [ <host_spec> ]
  • listObj
      • - List the objects of the applications that are part of the specified, or focused, experiment.
      - The listing can be restricted with the use of a <host_file_rpt> specification.
      - Any component described in the <host_file_rpt> specification must be part of the selected experiment.

      <ListOf_filename> = listObj [ <expId_spec> ] [ <host_file_rpt> ]
  • listPids
      • - List running processes info on a specific machine or on a specific cluster.
      - The absence of any -h <hostname> specification will cause all pids on all systems in the cluster to be listed.
      - The default cluster contains only localhost and must be specified by using -h localhost.
      - Use of the -f <filename> option will result in only the pids associated with that <filename> being listed for the selected hosts.
      - Mostly as ps(1), but output will have filename indication and/or rank, process, or thread form.
      - If -mpi is specified, the output will take the MPI rank format.

      <ListOf_pidname> = listPids [ <host_file> ] [ -mpi ]
  • listSrc
      • - List the source that was used to create the applications that are part of the specified, or focused, experiment.
      - The listing can be restricted with the use of a <host_file_rpt> specification.
      - Any component described in the <host_file_rpt> specification must be part of the selected experiment.

      <string> = listSrc [ <expId_spec> ] [ <host_file_rpt> ] [ <linenumber_spec> ]
  • listMetrics
      • - Retrieve the metrics that can be produced by an experiment. This is a list of the types of information that an experiment can gather.
      - The absence of any option will cause all the metrics for all the experiments that have been defined by the user, through the use of expCreate commands, to be listed.
      - The use of "focus" will cause metrics for only the focused experiment to be listed.
      - The use of <expId_spec> will cause the metrics for only that experiment to be listed.
      - The use of "all" will cause all the metrics for all of the available performance measurement utilities, that can be used to define experiments, to be listed.

      <ListOf_expMetric> = listMetrics [ focus || <expId_spec> || all ]
  • listParams
      • - Retrieve the parameters, to a specified experiment, that can be changed by the user.
      - The absence of any option will cause all the parameters of all the experiments that have been defined by the user, through the use of expCreate commands, to be listed.
      - The use of "focus" will cause parameters for only the focused experiment to be listed.
      - The use of <expId_spec> will cause the parameters for only that experiment to be listed.
      - The use of "all" will cause all the parameters for all of the available performance measurement utilities, that can be used to define experiments, to be listed.

      <ListOf_expParam> = listParams [ focus || <expId_spec> || all ]
  • listReports
      • - Retrieve the reports that can be produced by an experiment.
      - The absence of any option will cause all the possible reports for all the experiments that have been defined by the user, through the use of expCreate commands, to be listed.
      - The use of "focus" will cause reports for only the focused experiment to be listed.
      - The use of <expId_spec> will cause the reports for only that experiment to be listed.
      - The use of "all" will cause all the reports for all of the available performance measurement utilities, that can be used to define experiments, to be listed.

      <ListOf_viewType> = listReports [ focus || <expId_spec> || all ]
  • listBreaks
      • - List the breakpoints that have been set by the user.
      - The absence of any option will cause all the breakpoints for the focused experiment to be listed.
      - The use of <expId_spec> will cause all the breakpoints for the specified experiment to be listed.
      - The use of all will cause all the breakpoints for all of the experiments defined by the user, through the use of expCreate commands, to be listed.

      <ListOf_breakId> = listBreaks [ <expId_spec> || all ]

    Type, Status, Parameter Definition Section


  • expMetric
    •  is a result that an experiment can return. It could be a full report, a value or a set of values.

      <metricname> = { ? }

      <expMetric> = [<expType>::]<metricname>

  • expParam
    •  is the experiment parameters correspond to items in the experiment class that can be changed by the user or a tool component.

      <paramname> = { displayMode || displayMedia || displayRefreshRate || expSaveFile || samplingRate ? }

      <expParam> = [<expType>::]<paramname>

      - If <expType> is specified, then only that collector's parameter is updated.
      - If <expType> is NOT specified, then all parameters matching the parameter name are updated, regardless of the collector name, for that experiment .
  • expType
    • is the experiment type designator. Using this in the expCreate or expAttach command fills in the default values for the experiment.
      <predefined_exp> = { pcsamp || usertime || mpi || fpe || hwc || io || ? }

      <user_defined_exp> = <filename>

      <expType>> = [ <predefined_exp> || <user_defined_exp> ]

      <expType_list> = <expType> [ " ," <expType> ] ...

  • host_file_rpt
    •  represents the host file and machine along with a specific process, thread, or rank identifier.
      <IP_Address> = <int>.<int>.<int>.<int>.

      <rankname> = <int>

      <threadname> = <int>

      <pidname> = <int>

      <filename> = <string> || " <string> "

      <hostname> = <string> || <IP_Address>

      <rank_range> = <rankname> [ : <rankname> ]

      <rank_list> = <rank_range> [ , <rank_range> ] ...

      <thread_range> = <threadname> [ : <threadname> ]

      <thread_list> = <thread_range> [ , <thread_range> ] ...

      <pid_range> = <pidname> [ : <pidname> ]

      <pid_list> = <pid_range> [ , <pid_range> ] ...

      <host_spec> = -h <hostname>

      <host_file> = [ <host_spec> ] -f <filename>

      <file_list> = <filename> [ , <filename> ] ...

      <host_list> = <hostname> [ , <hostname> ] ...

      <host_file_rpt> = [ -h <host_list> ] [ -f <file_list> ] [ -p <pid_list> ] [ -t <thread_list ] [ - r <rank_list ]

      <host_file_rpt_list> = <host_file_rpt> [ , <host_file_rpt> ] ...
  • ListOf_
    • xxx     indicates a list of type xxx items.

  • viewType
    • is the output view display/report type designator. This indicates the format of the output display that the performance data is needed for.

      <viewType> = { vTopN || vExclTime || vIO || vFpe || vHwc || ? }

    More detailed example scenario 1 to show command usage

    This scenario is the same as the simple scenario 1 above but this one has more comments interspersed between the commands. It has a bit more information about what is going on under the hood.

    #User launches Open/SpeedShop tool from a window. The CLI tool will come up first and then if the GUI option is not turned off the GUI will launch.
    #The GUI will send low level commands such as:

    # create an experiment to do pcsampling of the executable named a.out on host1

    expCreate -h host1 -f a.out pcsamp

    "1"

    # expCreate will cause the initial default values of the specified experiment to be instantiated and the proper collector to be associated with the newly created experiment. The "1" below the command is the <expId> for the new pcsamp experiment.

    # expCreate also will "under the hood" poll the Framework to give us a list of the ranks, processes, and threads associated with a.out that are currently known to the FrameWork. expCreate will create a thread group including all the ranks, processes, and/or threads from the Application Object and associate the new thread group with the experiment that has just been created.

    # listPids will list the ranks, processes, and threads on the specified list of hosts.

    listPids -h host1

    # The GUI presents the user a panel from which he/she chooses which ranks, processes, and/or threads he/she wants included in the experiment. The GUI sends down expAttach commands to add them to the thread group or expDetach to subtract them from the thread group that will be operated on by the experiment. Naturally if the user wants to operate on all the ranks, processes, and/or threads no additional commands to select or deselect are necessary.

    # In this example scenario we only want to look at ranks 2 and 5, so we can use the expDetach command to remove all the ranks in the thread group associated with this experiment.

    expDetach -h host1 -r *

    # Now we use the expAttach command to add the ranks 2 and 5 into the thread group.

    expAttach -h host1 -r 2, 5

    # An alternative would have been to specify the ranks on the original expCreate command:

    expCreate -h host1 -f a.out -r 2, 5 pcsamp

    # listParams retrieves the list of parameters that may be changed by the user. The GUI and CLI tools need to display this to the user in order for the user to have the option of tuning the experiment to their specific needs.

    listParams

    1 not executing

    pcsamp::samplingRate=10
    displayMode=AtCompletion
    displayMedia=GUI
    displayRefreshRate=0
    <other default values>

    # We assume the user did not change the parameters in this part of the scenario because no expSetParam command is created.

    # listExp lists the active experiments to the screen

    listExp

    1 not executing

     -h host1 -f a.out -r 2,5 pcsamp

    # expSetParam defines arguments for the specified experiment. Here the "-x 1"" indicates the experiment identifier, however we can omit this argument as the tool will have a concept of default experiment. The experiment identifier is created by expCreate. The identifier created becomes the default current experiment. So commands following the expCreate will default to using the current experiment identifier as their experiment.

    # For the expSetParam the expId is optional because the default focused experiment was set to "-x 1" by the expCreate command.

    expSetParam samplingRate=20

    expSetParam displayMode=realtime

    expSetParam displayMedia=GUI

    expSetParam displayRefreshRate=60

    # Run the application and the performance data collection utilities. experiment was created this command will also cause the instrumentation to be inserted into the application e.g. instruments the thread group. (the set of ranks, processes, and/or threads that are associated with the experiment). The experiment is starts the gathering of performance data as the application is started running.

    expRun
    • Now we want to stop the application and change the sampling rate
      •

    # We use the expPause command we halt/pause the performance data collection and the application.

    expPause -x 1

    # Here we use expSetParam to replace the existing sampling rate for experiment "-x 1" with the new value.

    expSetParam -x 1 samplingRate=50

    # If we used expPause above we must do an expCont here to restart the application and to reengage the gathering of performance data.

    expCont -x 1

    # The listExp command is presented here, only to show the it can be used at any point in a sequence to give experiment data back to the GUI or the CLI tools.

    listExp -x 1

    1 executing, data available

     -h host1 -f a.out -r 2,5 pcsamp
    pcsamp::samplingRate=50
    displayMode=realtime
    displayMedia=GUI
    displayRefreshRate=60
    <other default values>

    • Now stop the experiment and view the results.
      •

    # We use the expPause command to stop the application. The focused experiemnt is "-x 1", so it does not need to specified in the command.

    expPause

    # The possible report types for this experiment are obtained. The GUI may want to present the user with a list of available views for this experiment (.e.g. in a menu item list. ).

    expGetMetrics
    "vTopN, vExclTime, vIO, vFpe, vHwc"

    # expView causes much processing behind the scenes. This command is asking for the data to create a top "n" display and will open a gui window to display the result.

    expView -x 1 -gui vTopN
    • Terminate the performance analysis session.

    # Close Open/SpeedShop. The data that was collected for the experiment is discarded, the executable modules that were attached to the experiment are terminated and all memory is reclaimed by the system.

    exit

    More detailed example scenario 2 to show command usage

    This scenario is the same as the simple scenario 2 above but this one has more comments interspersed between the commands. It has a bit more information about what is going on under the hood.

    #User launches Open/SpeedShop tool from a window. The CLI tool will come up first and then if the GUI option is not turned off the GUI will launch.

    # expCreate will cause the initial default values of the specified experiment to be instantiated and the proper collector to be associated with the newly created experiment. The "1" below is the <expId> for the new pcsamp experiment.

    # Since there is no executable specified on the expCreate, the experiment can not be Run but will act as a place holder for further commands.

    expCreate pcsamp

    "1"

    # expCreate creates a second experiment, this one for io performance anlysis. The default experiment is now "2".

    expCreate io

    "2"

    # expCreate creates a third experiment, this one for fpe. The default experiment is now "3".

    expCreate fpe

    "3"

    # This expAttach command will associate the a.out, b.out, and c.out on host1 with experiment 1 as specified by the "-x 1" annotation.

    expAttach -x 1 -h host1 -f a.out, b.out, c.out

    # This expAttach command will associate the a.out and c.out on host1 with experiment 2 as specified by the "-x 2" annotation.

    expAttach -x 2 -h host1 -f a.out, c.out

    # This expAttach command will associate the b.out and c.out on host1 with experiment 3. This is not specifically called out via the "-x 3" annotation because experiment "3" is the default experiment.

    expCreate -h host1 -f b.out, c.out

    # expCreate adds a collector to the third experiment, specifically this command adds a collector for malloc performance analysis in addition to the fpe collector. The default experiment is still "3".

    expCreate -x 3 malloc

    # listParams retrieves the list of parameters that may be changed by the user. The GUI and CLI tools need to display this to the user in order for the user to have the option of tuning the experiment to their specific needs. This command gets the parameters for experiment "1" which is the pc sampling experiment.

    listParams -x 1

    1 not executing

    pcsamp::samplingRate=10
    displayMode=AtCompletion
    displayMedia=GUI
    displayRefreshRate=0
    <other default values&glt;
